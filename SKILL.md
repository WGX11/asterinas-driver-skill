# Asterinas Driver Dev

## 何时使用

- 当你在实现或修改 Asterinas 驱动相关代码
- 当你需要知道某类驱动问题通常先检查什么，再决定怎么写
- 当你需要总结“为什么仓库里这样写”，而不是照着某次补丁抄代码

## 这份技能主要提供什么

1. 先看什么，再动手。
2. 哪些判断顺序不能乱。
3. 哪些约束必须一直带着。
4. 哪些常见写法其实是在规避驱动里的典型坑。
5. 哪些做法看起来能跑，实际上容易把同类题带偏。

## 使用方式

1. 先定位这次要改的是哪一类驱动问题。
2. 优先从这里找可复用的做法、判断顺序和反模式。
3. 如果这里还不够，再去看 `references/` 里的补充说明。
4. 把这里的内容当成“如何思考和如何落地”的总结，不要当成源码片段仓库。

## 这份技能是怎么维护的

- 这份技能的维护工作由 Supervisor 通过 `asterinas-skill-creator` 完成。
- 修改前会先保存旧版本，再直接更新这里。
- 更新后会先看当前失败题所属类型有没有变差，再看全量回归结果。
- 维护流程本身不放在这里展开，避免把 Student 会看到的正文写成运维手册。

## 错误处理

- 如果局部验证失败，不自动回滚；继续改当前版本，或恢复到备份版本。
- 如果全量回归失败，不自动回滚；继续修改，直到这版不会拖累其他 driver 题。
- 如果内容开始变成源码摘抄，立刻改写成“什么时候这样写、为什么这样写、不要怎么写”。
- 如果正文过长，把长解释下沉到 `references/`。

## 示例用法

```text
用户: 这个 console 类题目老是错，我需要知道这类驱动代码通常先看什么、怎么判断、哪些地方不能漏。
动作: 先从总驱动技能里找这一类问题的可复用做法和反模式，再决定代码怎么改。
预期输出: 一套能跨题复用的驱动写法，而不是一段照抄的源码。
```

## 反模式

- 不要把这份技能写成某一道题的标准答案。
- 不要大面积粘贴源码、完整函数或整段 diff。
- 不要跳过局部验证和全量回归。
- 不要把只对一个补丁成立的偶然写法包装成通用规律。
